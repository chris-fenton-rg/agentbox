Every time you launch an agent, AgentBox teleports your project into its own
isolated Linux computer — a **box**. Inside, the agent has root: it installs
packages, runs servers, runs Docker, and commits freely — and it still can't
touch your laptop. Run several boxes in parallel, switch between them in under a
second, and nothing reaches a remote until you approve it.

Six ideas carry the whole tool, and they're easier to feel than to read. Start
here, then follow the links into each one.

## A full computer per agent — and it can't break your machine

A box is a real, disposable Linux machine, one per agent run. The agent gets
root and a full toolchain — `apt install`, run a database, even run Docker inside
the box — so it's never boxed in by a thin container. When you're done, throw it
away; your laptop never saw any of it.

```bash
agentbox create -n alpha     # spin up a fresh box
agentbox destroy alpha       # throw it away
```

Because the box is sealed off from your host, you don't have to babysit the
agent. Let it run full-tilt — the blast radius is the box, and the box is
disposable.

<Callout title="WHY">
Auto-approve is safe here. The agent can't reach your files, your keys, or your
network — so it runs without stopping to ask, and you keep a clean machine.
</Callout>

The box keeps everything it built until you `destroy` it — `stop` and `pause`
preserve it. Lifecycle and resource detail: [Local Docker](/docs/local-docker).
Running Docker inside: [Docker-in-Docker](/docs/docker-in-docker).

## Run agents in parallel, switch in under a second

One command creates a box, starts the agent, and attaches you. Run Claude Code,
Codex, and OpenCode — each in its own box — and let them work at the same time.

```bash
agentbox claude -n feature        # Claude Code on one box
agentbox codex -n refactor        # Codex on another, in parallel
agentbox claude attach feature    # jump back to the first
```

Each agent runs in a detachable tmux session — press `Ctrl+a d` to detach and it
keeps working. Switching is instant: idle boxes are **paused**, not stopped, so
they sit at zero CPU with memory still mapped. `unpause` resumes in well under a
second and language servers pick up mid-thought.

<Callout title="TIP">
Detach with `Ctrl+a d`, fire up a few more boxes, and let them run. Reattach to
whichever one you want — switching pauses the one you leave and resumes the one
you open.
</Callout>

Per-agent detail and auth: [Run an agent](/docs/run-an-agent). Many boxes and
queued background runs: [Background &amp; parallel](/docs/background-and-parallel).

## Start warm with checkpoints

The first box in a project installs dependencies and warms its caches. Capture
that with a **checkpoint** — a snapshot of `/workspace` with its `node_modules`,
build caches, and in-box `.env` files — and every new box boots from it in about
a second instead of starting from bare code.

```bash
agentbox checkpoint create alpha --set-default   # save the warm state as the project default
agentbox create -n warm                          # new box, already warmed up
```

The usual move: checkpoint right after the setup wizard finishes or a PR merges,
so the whole team's boxes start ready. Full mechanics:
[Checkpoints &amp; pausing](/docs/checkpoints-and-pausing).

## Local or in the cloud — same command

The default box is a local Docker container: fast, free, on your own machine.
When a run outgrows the laptop — heavier builds, a teammate who needs to attach,
or a public URL to share — point the same commands at a cloud with one flag. The
clouds hand back **public HTTPS preview URLs**, so the agent's dev server is one
click away.

| Provider | Where the box lives | Reach for it when |
| --- | --- | --- |
| `docker` (default) | Local Docker container | Fast, free, fully local — the right default. |
| `hetzner` | A Hetzner VPS, one per box | You want a real VM you control — root, full kernel, your region. |
| `daytona` | A managed Daytona sandbox | The work outgrows the laptop or a teammate needs to attach. |
| `vercel` | A Firecracker microVM | Fast snapshots, public URLs, free pause/resume (no in-box Docker). |

```bash
agentbox claude --provider hetzner     # run this agent on a Hetzner VPS
agentbox config set box.provider daytona   # or pin a default for the project
```

<Callout title="NOTE">
Each cloud needs a one-time `agentbox prepare --provider <name>` and a login
(`agentbox hetzner login`, etc.). Docker needs neither. One page each:
[Local Docker](/docs/local-docker), [Hetzner](/docs/hetzner),
[Daytona](/docs/daytona), [Vercel](/docs/vercel).
</Callout>

## Your real project, isolated git

A box doesn't start from `HEAD` — it starts from what you actually have.
AgentBox replays your uncommitted work (stash plus untracked files) into the
box, so the agent picks up exactly where you left off.

Inside, `/workspace` is a git worktree on a fresh per-box branch
`agentbox/<box-name>`, created against your **real** `.git`. The agent's commits
land in your repo immediately — but on that isolated branch, never on your
checked-out branch or working tree. Build artifacts (`node_modules`, `.next`,
`target`) live in the box and are rebuilt inside Linux, so host binaries never
leak in.

```bash
agentbox open mybox     # pull /workspace to a host folder and open it
```

How projects get in: [Teleport a project](/docs/teleport-a-project). Moving work
back and pushing: [Sync &amp; git](/docs/sync-and-git).

## Credentials never leave your machine

Boxes carry no SSH keys and no git credentials by design. So how does `git push`
work? Through the **host relay** — a small host process, started on your first
run and reused by every box. The agent asks the host to push, open a PR, or open
a URL; the host does it with **your** identity (SSH agent, `~/.gitconfig`, `gh`
auth) and streams the result back.

```console
$ agentbox-ctl git push
# the host relay runs `git push` as you — no credentials ever enter the box
```

<Callout title="WHY">
This is the safety model in one line: the agent runs unsupervised inside a sealed
box, and every push to a remote happens on your machine, as you, only when you
allow it. Nothing secret crosses into the box.
</Callout>

Push, pull, PRs, and approvals: [Sync &amp; git](/docs/sync-and-git).
